## Codebase Patterns
- Electron main process is `electron/main.cjs` (must be `.cjs` because `"type": "module"` in package.json); `"main"` in package.json points to it; server scripts still use explicit `src/index.js` path
- Electron preload script is `electron/preload.cjs`; exposes `window.electronApi` with 7 methods via `contextBridge.exposeInMainWorld`; all IPC calls use `ipcMain.handle`/`ipcRenderer.invoke` pattern with channel whitelist and origin validation (`validateSender` checks `event.senderFrame.url`)
- Electron app menu is in `electron/menu.cjs` (`buildAppMenu(mainWindow)` returns a `Menu`); system tray in `electron/tray.cjs` (`createTray(mainWindow)` / `destroyTray()`)
- Electron spawns server via `spawn(process.execPath, [serverScript], { env: { ELECTRON_RUN_AS_NODE: '1', ... } })` â€” this makes the Electron binary behave as Node.js for child processes
- electron-builder config is `electron-builder.config.cjs`; build output goes to `dist/`; icons in `build/` (icon.png, icon.ico) and `build/icons/` (sized PNGs for Linux)
- `electron-window-state` must be in `dependencies` (not devDependencies) so electron-builder bundles it â€” it's required at runtime by `electron/main.cjs`
- Icon generation: `node scripts/generate-icons.js` creates all icon sizes from scratch using raw PNG/ICO encoding (no external image libraries)
- Build scripts: `build:win` (NSIS + zip), `build:mac` (DMG), `build:linux` (AppImage + deb + rpm); `build` runs all platforms
- The `data/` directory (CA certs) is NOT bundled â€” the app generates CA certs at runtime on first launch
- Server port is auto-detected via `net.createServer()` on port 0; auth token generated with `crypto.randomBytes(32).toString('hex')` and passed via `AUTH_TOKEN` env var
- Design tokens are CSS custom properties in src/ui/styles.css, organized by theme (:root = dark, [data-theme="light"], [data-theme="high-contrast"])
- Reference source of truth for all design tokens: httptoolkit-ui/src/styles.ts
- Box shadows use `rgb(0 0 0 / var(--box-shadow-alpha))` so they auto-adapt to themes (dark=0.4, light=0.3, high-contrast=0.1)
- Font size tokens follow naming: `--font-size-{small-print|text-input|text|sub-heading|heading|large-heading|loud-heading|screaming-heading}`
- Link colors use `--link-color` and `--visited-link-color` variables (theme-aware)
- The project is vanilla JS (no framework, no TypeScript) â€” "typecheck" means verifying `node --check` passes on JS files
- All border-radius=4px, resizer=11px, sidebar=75px, header/footer=38px, row=32px
- Phosphor Icons loaded via `<link rel="stylesheet" href="/vendor/phosphor/regular/style.css">` â€” served from node_modules via Express static at `/vendor/phosphor`
- Phosphor icon class format: `<i class="ph ph-icon-name"></i>` (regular weight), size controlled via `font-size` CSS
- Sidebar icons: Intercept=ph-plug, View=ph-magnifying-glass, Mock=ph-pencil-simple, Send=ph-paper-plane-tilt, Settings=ph-gear
- All action/button icons use Phosphor `<i class="ph ph-icon-name">` elements (not inline SVGs). Only interceptor card brand icons (Chrome, Firefox, etc.) and decorative empty-state illustrations remain as SVGs.
- Traffic source icons: Globe, Terminal, Cube (Docker), GearSix (mock), FolderOpen (import), LockSimpleOpen (tls-error)
- CSS `.source-proxy i.ph`, `.source-mock i.ph`, `.source-import i.ph` set per-source icon colors
- Mock rule action Phosphor icons: FloppyDisk (save), PencilSimple (edit), ToggleRight/ToggleLeft (enable/disable), Tag (rename), CopySimple (clone), TrashSimple (delete)
- Pin icon uses PushPin with `transform:rotate(45deg)` when unpinned, `transform:none` when pinned
- For dynamic icon switching (e.g. togglePause), update `btn.innerHTML` with Phosphor `<i>` elements
- Monaco Editor served via Express static at `/vendor/monaco` from `node_modules/monaco-editor/min`
- Monaco loaded via AMD loader: `<script src="/vendor/monaco/vs/loader.js">` + `require.config({ paths: { vs: '/vendor/monaco/vs' } })`
- Use `createMonacoEditor(containerId, options)` to create editors â€” returns a Promise<editor>
- Monaco custom themes: `httptoolkit-dark` and `httptoolkit-light` â€” auto-synced via `setTheme()`
- `monacoReady` promise resolves when Monaco API is available; `monacoApi` holds the global reference
- `monacoInstances` array tracks all active editors for theme switching and cleanup
- Editor auto-resizes via ResizeObserver; auto-cleans up when container is removed from DOM
- Body viewers use dual-container pattern: `id="*-monaco"` (Monaco) + `id="*-fallback"` (pre); toggled by `isMonacoViewMode(mode)` â€” used by both request body (`reqBody-monaco`) and response body (`resBody-monaco`)
- `activeBodyEditors` tracks live Monaco body editors by container id; call `disposeBodyEditor(id)` before DOM replacement
- Non-Monaco body view modes: hex (textToHex), decoded (key-value grid), image (placeholder) â€” all render in the fallback `<pre>`
- Send page body editor: `sendBodyEditor` variable holds the editable Monaco instance; use `getSendBodyValue()` / `setSendBodyValue()` to read/write; `updateSendBodyLanguage()` syncs language from format dropdown
- `monacoApi.editor.setModelLanguage(model, lang)` switches language without recreating the editor instance
- Traffic list uses virtual scrolling: only ~60 DOM rows rendered at a time via spacer `<tr>` elements
- Virtual scroll constants: `VS_ROW_HEIGHT=32`, `VS_BUFFER=15`, `VS_HEADER_HEIGHT=38` (sticky thead)
- `renderVirtualRows()` is the lightweight scroll-triggered renderer; `renderTraffic()` handles full updates (counts, empty state, auto-scroll)
- Spacer pattern: top `<tr class="vs-spacer">` + visible rows + bottom `<tr class="vs-spacer">` â€” total height = totalRows * 32
- `vsForceRender = true` forces re-render even when scroll range unchanged (used after selection changes)
- `scrollRowIntoView(index, alignment)` scrolls a row into the viewport; supports `'center'` alignment
- Scroll handler uses `requestAnimationFrame` to batch scroll events for 60fps performance
- Row HTML uses `data-id="${req.id}"` attribute for identification instead of parsing onclick strings
- Scroll position persistence uses localStorage keys `trafficScrollTop` and `trafficAutoScroll`; restored via `restoreTrafficScrollPosition()` in both `switchPanel()` and `navigateFromHash()`
- WebSocket frame parser: `WsFrameParser` class in `src/proxy/ws-frame-parser.js` â€” streaming parser with `push(chunk)` and `onFrame` callback
- WS frame events emitted with `protocol: 'ws-frame'`, `parentId` linking to the parent WS connection, and `direction: 'client'|'server'`
- WS frame opcodes: `WS_OPCODE` constants and `WS_OPCODE_NAMES` map in ws-frame-parser.js; `parseClosePayload()` extracts close code + reason
- Frame parsing is non-blocking: raw bytes are always forwarded even if parser throws; parser errors don't break the proxy
- WS frame sub-rows: `wsFramesByParent` map groups frames by parentId; `wsExpandedConnections` set tracks expanded WS connections; `applyFilter()` rebuilds the map and inserts frames inline into `filteredRequests`
- WS frame rows use `.ws-frame-row` CSS class (28px height, indented, lighter bg); parent WS rows show expand toggle + frame count badge
- Stream message list card in WS detail view renders all frames chronologically with clickable rows that call `selectRequest(frameId)`
- WS frame detail view: text frames render payload in Monaco (auto-detects JSON), binary frames render hex dump, close frames show code + reason
- HTTP/2 upstream: `_getH2Session(hostname, port)` returns cached h2 session or null (blacklisted/unsupported); `_makeH2Request()` sends request via h2 stream; sessions cached in `_h2Sessions` Map per `host:port`, evicted after 60s idle or on error/close/goaway
- H2 upstream is skipped when `this.upstreamProxy` is configured (upstream proxy uses CONNECT tunneling which is always h1.1)
- `_h2Blacklist` Set remembers origins that failed h2 connect to avoid repeated 5s connect timeouts
- All three HTTPS forwarding paths (_handleTlsConnection, _handleHttp2Connection stream, _handleHttp2Connection h1 fallback) try h2 upstream first with h1.1 fallback
- TLS error events: `protocol: 'tls-error'`, `source: 'tls-error'`, include `error` (message) and `errorCode` (e.g. `ERR_SSL_WRONG_VERSION_NUMBER`) fields
- Tunnel events: `protocol: 'tunnel'`, `source: 'tunnel'`; emitted for TLS passthrough (on close) and MITM connections with no HTTP request within 5s
- TLS error and tunnel rows use `.tls-error-row` and `.tunnel-row` CSS classes (italic, 28px height, centered text) â€” distinct from standard 32px rows
- Row marker colors: TLS error = `#ce3939` (red), tunnel = `#888` (grey)
- Special row types (tls-error, tunnel) are handled as early returns in `buildRowHtml()` before the standard row logic, same pattern as `ws-frame`
- Expandable interceptor cards: `EXPANDABLE_INTERCEPTORS` Set defines which interceptors expand in-grid; `expandedInterceptorId` tracks the currently expanded card; `handleExpandableCardClick()` activates + expands; `collapseInterceptorCard()` collapses
- Expanded card CSS: `.intercept-card.expanded` spans 2 columns + 2 rows; `.intercept-card-close` is the X button; `.intercept-card-config` contains the config content
- Config renderers: `renderDockerConfig(container)` shows docker run/compose commands; `renderTerminalConfig(container)` shows shell-tabbed env var commands with copy-to-clipboard
- Config code blocks: `.config-code-block` has click-to-copy behavior via `copyConfigCode(el)`; `.config-tabs` + `.config-tab` for shell switching
- Android ADB interceptor: `_getConnectedDevices()` parses `adb devices -l`; emulators use `10.0.2.2` as host IP, physical devices use LAN IP via `os.networkInterfaces()`
- Interceptor interface: `isActivable()`, `isActive()`, `activate(proxyPort, options)`, `deactivate()`, `toJSON()` â€” register via `this._register()` in InterceptorManager constructor
- Android ADB UI: `renderAndroidConfig()` renders device list from `expandedInterceptorMetadata.devices`; `activateAndroidDevice(deviceId)` calls activate with deviceId; `refreshAndroidDevices()` re-fetches device list
- For interceptors that need dynamic metadata refresh (like android-adb, jvm), `handleExpandableCardClick` always calls activate even when already active
- JVM interceptor uses `jps -v` for process discovery, Java Attach API for agent loading, and falls back to proxy flag instructions if attach fails
- JVM process list UI follows the same pattern as Android device list: `renderJvmConfig()`, `activateJvmProcess(pid)`, `refreshJvmProcesses()` â€” all mirroring the Android equivalents
- Mock rules use a draft model: edits stored in `mockDraftRules` Map (keyed by rule ID), new drafts tracked in `mockNewDraftIds` Set; `saveMockRule()` saves to draft, `saveAllMockRules()`/`saveOneMockRule()` persist to server
- Draft rules have orange left border (`.mock-rule-draft`) and glow; `updateMockSaveButtons()` shows/hides Save All + Revert buttons and unsaved badge
- `_applyDraftToLocal(ruleId, draft)` overlays draft data onto the local `mockRules` array â€” called from `loadMockRules()` to preserve drafts after server reload
- Navigation away from Mock page with unsaved drafts triggers a `confirm()` dialog in `switchPanel()`
- Inline editing pattern: `mockRenamingRuleId` state + `startInlineRename()`/`confirmInlineRename()`/`cancelInlineRename()`; input rendered in `renderMockRuleRow()` with Enter/Escape/blur handlers; use `setTimeout(focus, 0)` after re-render
- Transform perspective: `_transformPerspective` state ('original'|'transformed'|'client'|'server'); `getEffectiveRequest(req)` overlays `req.originalRequest` data when viewing original perspective; `switchTransformPerspective(value)` re-renders detail cards
- Proxy emits `originalRequest` (with method, url, headers, body) and `transformedBy` (rule title/id) on mock-matched requests whose preSteps modified method/URL/headers
- Transform card uses `.transform-card` CSS class with pop-color left border and 6% tinted background
- Auto-updater: `electron/updater.cjs` exports `initAutoUpdater(win)` and `stopAutoUpdater()`; uses `electron-updater` with GitHub Releases as default provider; update feed URL overridable via `UPDATE_URL` env var
- Preload IPC pattern: `ALLOWED_INVOKE_CHANNELS` for request/response IPC, `ALLOWED_ON_CHANNELS` for push events from main process (used by updater status notifications)
- UI listens for updater events via `window.electronApi.onUpdaterStatus(callback)` â€” returns unsubscribe function; all auto-updater UI is gated behind `window.electronApi` existence check
- ARIA accessibility: sidebar uses `role="tablist"` with `role="tab"` items and `role="tabpanel"` panels; `switchPanel()` syncs `aria-selected`; traffic table has `role="grid"` with `aria-rowcount`, rows have `role="row"` + `role="gridcell"` cells, `id="row-{reqId}"` for `aria-activedescendant`; collapsible detail cards get `aria-expanded` toggled in `toggleCardCollapse()`; mock rules/groups get `aria-expanded` in their render functions; send tabs use `role="tablist"` + `role="tab"` with `aria-selected`
- Skip-to-content: `<a href="#main-content" class="skip-to-content">` is the first focusable element in `<body>`; CSS hides it offscreen until `:focus`
- Matcher types are defined in `MOCK_MATCHER_GROUPS` (grouped with optgroups) and `MOCK_MATCHER_TYPES` (flat, derived via `.flatMap()`); 21 total matchers across 4 groups: Basic (wildcard, method, host, path), URL (hostname, regex-path, regex-url, url-contains, query, exact-query, port, protocol), Headers (header, cookie), Body (body-contains, json-body-exact, json-body-includes, regex-body, raw-body-exact, form-data, multipart-form-data)
- `host` matcher uses `new URL(url).host` (includes port); `hostname` matcher uses `new URL(url).hostname` (no port) â€” both support wildcard prefix matching
- Multipart form data matcher parses boundary from Content-Type header and splits body on `--boundary` to find field names via Content-Disposition regex
